CCR Platform - Product & Architecture

The under-the-hood companion to the testing guide: who can do what, how access is governed, what runs where, and the engineering decisions behind it. The guide tells you how to use the platform; this page tells you how it works.

1. Access model: four tiers + break-glass

Every account has one of four roles. New sign-ups land as external user unless an invite link or a pre-assigned role says otherwise (see §2). Roles are managed entirely in the app, on the /admin page - no server configuration involved.

Two deliberate asymmetries: verification belongs to maintainers - the trail then names the responsible RA, not whichever admin clicked - while oversight belongs to the PI, who sees everything (including the audit trail) but delegates the hands-on QA. And nobody can change their own role, so no session can quietly climb the ladder.

Break-glass: the ADMIN_EMAILS environment variable A short server-side allowlist holds the same powers as the PI role. It exists for exactly two reasons: seeding the very first PI (someone has to grant the first role) and recovering a locked-out lab. Day-to-day it should be one entry - the developer - and everything else lives in the app.

2. Access before first sign-in

Three ways someone has the right access before they ever log in, in increasing order of privilege:

  1. Anonymous use - no account at all: 3 runs/day, small files, uploads deleted right after analysis. The zero-commitment trial tier.
  2. Invite links - a signed URL an admin copies from /admin into Slack or an email. Whoever registers through it lands as a lab member (or external user). Bearer-style: anyone with the link can use it, so it only carries non-staff roles and expires after 7 days. Each link is tracked on /admin: revoke it any time (it stops working immediately) and see exactly which accounts signed up through it; creation, revocation, and every redemption are audited.
  3. Pre-assigned roles - a role bound to a specific email before the account exists. When that person first signs in - password or Google - they land at the assigned tier, staff roles included. This is how an external collaborator gets full credentials on day one: pre-assign maintainer to their email, send them the URL, done. Staff pre-assignments require PI (or break-glass) rights, are listed as pending until claimed, and can be removed any time before the claim.

3. How it works (architecture & data flow)

The whole system is deliberately simple: one application, a few managed services around it, and no third-party AI APIs — the embedding models run on our own server, so uploaded text never leaves the platform.

Where your data lives (and for how long)

  • Records (account, projects, run results and summaries) → the Postgres database. Persistent, backed up.
  • Uploaded files → object storage. Signed-in users' files persist; anonymous uploads are deleted the moment their analysis finishes.
  • Anonymous sessions are wiped entirely after 24 hours; nothing you upload anonymously is kept.
  • Reproducibility: every run also records the exact model version, an item-wording hash, and package versions, so the downloadable script reproduces the numbers on any machine.

A few engineering choices worth knowing

  • Same model for items and texts, always — mixing models makes scores non-comparable, so the platform records the model on every run and warns you.
  • MiniLM is the default because it is the reference model in the published CCR work; the stronger E5 models are available and clearly labelled.
  • Re-running a new construct on the same corpus is near-instant — the text embeddings are cached and reused (identical every time), so only the cheap similarity step repeats.
  • Warnings never silently "fix" your data — the platform surfaces issues (wrong language, truncation, duplicates) and lets you decide.
  • Admin power is layered, not global — see §1: operations are delegated to maintainers, escalation stays with the PI, and one env variable exists purely as the fire escape.
